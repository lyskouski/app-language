## Root Cause - How the Fix Worked

### Original Problem
PyJNIus has `pyproject.toml` with `setuptools.build_meta` backend. When buildozer/p4a tried to build it using modern PEP 517 mechanism, it created an isolated environment where setuptools couldn't import properly.

### Solution Applied

1. **Custom PyJNIus Recipe** (`android_recipes/pyjnius/__init__.py`)
   - Removes `pyproject.toml` before build
   - Forces use of traditional `setup.py install`
   - Avoids isolated build environment

2. **Buildozer Configuration** (`buildozer.spec`)
   - Changed `p4a.setup_py = false` → `true`
   - Signals p4a to prefer setup.py over modern backends

3. **Dependency Pre-installation** (`.github/workflows/build.yml`)
   - Pre-installs setuptools, wheel, cython
   - Ensures dependencies available if isolation still occurs

### Why It Works
The custom recipe overrides the build process to use the **same methodology as other successful Python recipes in p4a**. By calling `setup.py install` directly with proper environment variables, it bypasses the problematic PEP 517 isolated build mechanism.
